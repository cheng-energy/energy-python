###### 网络的基础知识
- 网络：就是互联网，无数个计算机设备连接起来的全球性的网络基础设施
- IP地址：设备在网络的身份证，用来定义设备在网络中的地址
    - IPv4的地址：32位二进制（四段数字，每一段都是0-255）
    - IPv6地址：128位二进制
    - 平时看到的是10进制
    - 127.0.0.1是一个非常特殊的IP地址，表示的是本机地址（本地回环地址）--我的IP是127.0.0.1有本事就攻击我（手动狗头）
- 域名：IP地址不便于记忆，所以发明域名（网址），IP地址和域名是通过DNS(域名解析服务器)来将域名和IP地址映射，以此记忆与访问

    `localhost`是本机域名
- 端口：用来标识计算机设备上运行的程序（例如通过DNS得知百度的IP地址，但是百度有很多应用，百度网盘，爱奇艺，百度地图等，通过端口就可以定位到具体的应用程序）

    注意：`HTTP协议默认端口号是80，HTTPS协议默认端口号是443`

总结：
1. IP地址是用来定位计算机设备的
2. 域名是为了降低IP地址的记忆和书写创建的（要用DNS把域名IP进行映射）
3. 端口是确定访问的那个计算机设备启动哪个程序，每一个程序对应一个端口(端口的范围是0-65535)

##### HTTP协议的介绍
- 客户端和服务端连接成功之后就会涉及到数据的传输- 数据传输的规则（网络模型和网络协议）
###### 网络模型
- 国际ISO规定程序在网络通信中的模型和标准
    - OSI模型
        - 七层网络模型（应用层，表示层，会话层，传输层，网络层，数据链路层，物理层）
    - TCP/IP网络模型（可以理解位OSI的简化）
        - 四层网络模型
            - 应用层：HTTP,FTP,SMTP--将交互的数据按照协议进行封装（HTTP协议更多）
            - 传输层：UDP（速度快花销小，不保证数据的可靠）,TCP（确认数据的可靠）--通过端口等把数据包精准发送给对应的应用程序
            - 网络层：IP--通过IP把数据包发给对应的设备
            - 网络接口层：处理和硬件设备的关注
###### HTTP协议-超文本传输协议（规定客户端和服务端的传输规则）
1. 请求和响应都遵循相同规则才能读懂来自对方的数据

特点：
1. 文本格式，稳定性强
2. 请求响应模型，一次请求对应一个响应，先有客户端请求服务器端才会相应
3. 无状态，服务器端是不会记忆历史交互信息，每一次的请求和响应都是相互独立的


##### HTTP请求数据的格式
第一行：

请求行（请求方式，资源路径，协议）
```
请求方式
- GET：请求参数在请求行中，没有请求体。如/api/course?name=python&status-1。GET请求的参数大小在浏览器中是有限制的(?之后的就是请求参数，key=valus的形式)
- POST：请求参数在请求体中，POST请求大小是没有限制的(请求格式的最后面{}里面的内容)
```

第二部分：

以key：value的形式出现---请求头（key:value）

第三部分：

请求体（请求头和请求体有空行，GET请求没有请求体，POST请求可以没有请求体）
##### HTTP响应数据格式

响应行（协议，状态码）
- 200---成功
- 400---请求参数错误
- 404---请求资源不存在
- 500---服务器发生了不可预期的错误

响应头（key：value）

响应体（存放响应后的数据）

- 流式输出就是一个字一个字输出（在调用deep seek过程中会涉及）
- 非流式输出就是一下子输出
- curl是一个命令行工具（-H后面的是请求头）
- nodejs是一种代码
- json格式，类似于字典（curl{}里面的内容就是请求体），所有的key必须有双引号引起来。值可以是任意类型（对象，数字，字符串，布尔，列表）
```
"messages": [
          {"role": "system", "content": "You are a helpful assistant."},
          {"role": "user", "content": "Hello!"}]
```
其中，role system 就是系统提示词，告诉大模型自己，他是什么，他自己的身份和性格是什么样的（人设）

role user 就是这次交互我们发送过去的内容

大模型响应回来之后，role assistant就是大模型响应的，下面的content就是响应的内容

- 发送和响应都会消耗词元（token）

- 这样进行对话大模型是没有记忆的（每一次的请求和响应是相互独立的）

**会话历史滚雪球**：第二次发送需要把我们上一次发送的内容和AI大模型回答加入到messages中然后发送
```
"messages":[
    {"role":"system":},
    {"role":"user":},
    {"role":"assistent","content":},
    {"role":"user","content"},
]
```
- 通过上下文AI可以知道我们之前的会话记忆

##### 基于python代码调用大模型
```py
#调用deep seek
import os  # 官方提供内置库，用来获取操作系统信息--这里主要获得（获取一个环境变量）deep seek的APIkey
from openai import OpenAI
# client是基于OpenAI建立的客户端对象（DEEPSEEK_API_KEY就是环境变量的名字，值就是APIkey的值）--需要先创建一个环境变量
client = OpenAI(
    api_key=os.environ.get('DEEPSEEK_API_KEY'),
    base_url="https://api.deepseek.com")

response = client.chat.completions.create(
    model="deepseek-chat",
    messages=[
        {"role": "system", "content": ""},
        {"role": "user", "content": ""},
    ],
    stream=False
)

print(response.choices[0].message.content) # 与响应回的格式有关
```

### 提示词工程
- prompt：引导大模型进行内容生成的命令（一句话，一个问题等），引导AI思考，约束其输出范围，并明确你期望的结果
- 提示词工程:通过有技巧编写提示词，让大模型尽可能产生预期内容，这种持续性过程就是提示词工程
1. 定义大模型的角色和能力
2. 明确核心的请求和任务
3. 按步骤拆分复杂的任务
4. 指定风格和语气
5. 明确输出的格式（可以给出一个格式样例）
6. 提供输入输出的示例

注意：并不是说必须要有这6点

总结就是三点：角色，任务，要求。需求越明确结果越准确
